# 第九课作业：上下文工程实践

> **对应主课**：[第九课 — 上下文工程](./09-context-engineering.md)
>
> 本作业有三个核心练习和两个进阶挑战。核心练习对应课程的三个关键概念。

---

## 核心练习 1：给你的 Skill 写一个精准 Description

### 背景

课程讲到"描述是路由信号，不是营销文案"。现在你来实践。

### 任务

选一个你在用的 AI Agent 工具（可以是 Claude Code Skill、GPT Custom Instruction、Cursor Rule，甚至是一个 Makefile target），用 **"Use when / Not for / Output" 三件套** 重写它的描述。

**提示词**（给你的 AI 助手）：

> 我有一个 [工具/Skill/规则]，当前描述是：
>
> "[粘贴你现有的描述]"
>
> 请用以下模板重写，使它成为一个精准的路由信号：
>
> ```
> [一句话：做什么 + 交付物]。
> Use when: [3-5 个具体触发场景，用用户语言写]。
> Not for: [2-3 个排除场景，特别是和相邻工具的边界]。
> Output: [交付物的具体形态和约束]。
> ```
>
> 重写时注意：
> 1. "Use when" 用用户会说的话，不是内部术语
> 2. "Not for" 必须写——没有反例的路由器在边界请求时会困惑
> 3. "Output" 钉死交付物形态，减少漂移空间

### 验收标准

- [ ] 重写后的描述包含 Use when / Not for / Output 三部分
- [ ] Use when 至少 3 条，用用户视角的语言
- [ ] Not for 至少 2 条，明确和相邻工具的边界
- [ ] 找一个不了解这个工具的人，只看描述，判断"什么时候该用/什么时候不该用"——如果他判断对了，说明描述够精准

## 核心练习 2：给你的项目做一次"AC vs 原始需求"审计

### 背景

F041 教训：AC 是需求的有损压缩。12 条 AC 全绿，但丢了多项目管理、UI 描述、来源分类三个关键需求。

### 任务

选一个你正在做或最近完成的 feature，做一次审计：

1. **找到原始需求**——可能是 PRD、设计稿、用户故事、老板口头说的话（如果是口头的，先文字化）
2. **列出当前的验收标准**（无论叫 AC、DoD、Checklist 都算）
3. **逐条对照**，找出：
   - 原始需求中有，但 AC 中没有的项（遗漏）
   - AC 中有，但和原始需求表述不一致的项（偏差）
   - 原始需求中隐含但没显式写出的期望（隐性需求）

**提示词**：

> 这是我的 feature 原始需求：
> "[粘贴原始需求]"
>
> 这是我当前的验收标准：
> "[粘贴 AC 列表]"
>
> 请帮我做一次审计：
> 1. 列出原始需求中有但 AC 遗漏的项
> 2. 列出 AC 和原始需求表述不一致的项
> 3. 猜测用户可能有但没说出来的隐性期望
>
> 用表格展示：| 原始需求 | AC 中有没有 | 差距描述 |

### 验收标准

- [ ] 审计覆盖了至少一个真实 feature
- [ ] 产出了"遗漏/偏差/隐性需求"三类发现
- [ ] 至少发现 1 个遗漏项——如果一个都没有，大概率是审计不够仔细

## 核心练习 3：设计一次冷启动验证

### 背景

冷启动验证器的核心思想：找一个什么都不知道的角色，只给需求 + 交付物，让它判断"这是用户要的吗"。

### 任务

为你最近完成的一个 feature 设计一次冷启动验证。你需要：

1. **准备需求包**：≤10 行的用户原始需求摘要（不是 spec，不是 AC，是"用户的原话"或最接近原话的表述）
2. **准备交付物证据**：截图 / 录屏 / API 响应示例——无论什么形式，是最终用户能看到的东西
3. **设计 prompt**：让一个 AI Agent（或一个不了解项目的同事）只基于需求包和交付物证据，回答三个问题：
   - "用户的核心诉求是什么？"
   - "交付物满足了吗？缺什么？"
   - "如果你是用户，你会满意吗？"

**提示词**（给冷启动验证用的 AI）：

> 你是一个独立的验证者。你对这个项目一无所知。
>
> 以下是用户的原始需求：
> "[≤10 行需求摘要]"
>
> 以下是交付的产品 [截图/描述]：
> "[交付物证据]"
>
> 请回答：
> 1. 用户的核心诉求是什么？（用你自己的话概括）
> 2. 交付物满足了这些诉求吗？具体缺什么？
> 3. 如果你是这个用户，1-10 分你打几分？为什么？

### 验收标准

- [ ] 需求包 ≤10 行，用的是用户语言而非技术语言
- [ ] 交付物证据是用户可感知的形态（不是代码 diff）
- [ ] 冷启动验证者给出了至少一个你之前没想到的反馈

## 进阶挑战（选做）

### 挑战 1：为你的项目设计信息分层

参考我们的三层架构（始终在 ≤100 行 / 按需加载 ≤150 行 / 引用时加载），为你自己的项目设计一套信息分层：

1. **列出你当前的 Agent 配置文件**：CLAUDE.md / .cursorrules / Custom Instructions / 系统 prompt 等
2. **分类每一段内容**：始终需要 / 某些任务需要 / 很少需要
3. **设计分层方案**：哪些留在 Layer 0（始终加载），哪些拆到 Layer 1（按需），哪些降级到 Layer 2（引用时才加载）
4. **量化效果**：压缩前多少行/token？压缩后多少行/token？

**关键问题**：拆分后，模型在最常见的任务场景下，还能看到它需要的所有关键信息吗？

### 挑战 2：测试剧场审计

选一个有完整测试的 feature，做一次"测试剧场"审计：

1. **列出所有测试**，按类型分类：单元测试 / 集成测试 / E2E 测试
2. **问自己**：这些测试验证的是"代码逻辑正确"还是"用户体验正确"？
3. **找缺口**：有没有一个测试模拟"用户打开产品，完成核心流程"？如果没有——恭喜，你找到了测试剧场。
4. **补一个集成场景**：写一个测试（或手动检查流程），模拟用户的完整使用路径

**判断标准**：如果你删掉所有单元测试，只留集成/E2E 测试，你对交付质量还有信心吗？如果没有——说明你的测试金字塔需要重新平衡。

## 自检

完成以上练习后，问自己：

> "如果我的 Agent 下次交付了一个'100% pass，但不是我想要的'产品，我知道该在哪一层排查了吗？"

如果答案是"知道"，这节课就没白上。
