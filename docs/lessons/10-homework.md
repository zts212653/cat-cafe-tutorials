# 第十课作业：知识管理实践

> **对应主课**：[第十课 — 别让 AI 随地大小拉 markdown](./10-knowledge-management.md)
>
> 三个核心练习 + 两个进阶挑战。每个练习对应课程的一个关键概念。

---

## 核心练习 1：给你的项目做一次"蜘蛛网审计"

### 背景

F040 之前，问"F21 什么情况"要搜 85 个文件。你的项目有同样的问题吗？

### 任务

1. 选一个你正在做的项目，**列出所有文档存放位置**（README、docs/、wiki、Notion、飞书、local markdown……）
2. 选一个具体功能，试着回答："这个功能的需求从哪来？设计讨论在哪？实现计划在哪？Review 记录在哪？"
3. 记录你找到答案花了多少步骤（打开了几个工具/目录/文件）

**提示词**（给你的 AI 助手）：

> 帮我审计这个项目的文档分布。列出所有 docs/ 目录下的 markdown 文件，按以下维度分类：
>
> 1. 有 YAML frontmatter 的 vs 没有的
> 2. 有明确 feature 关联的 vs 孤立的
> 3. 活跃的 vs 可以归档的（最后修改超过 30 天）
>
> 输出一个表格，标注每个文件的状态。

### 验收标准

- [ ] 审计覆盖了至少 10 个文档文件
- [ ] 找到了至少 1 个"孤岛文档"（跟任何 Feature 都没有关联）
- [ ] 回答"X 功能什么情况"时，记录了具体的搜索步数

## 核心练习 2：给你的项目加 Frontmatter

### 背景

三个字段就能让 markdown 从"自由格式"变成"可机器索引"。

### 任务

从练习 1 的审计结果中，选 5-10 个没有 frontmatter 的文档，补上最小契约：

```yaml
---
feature_ids: []      # 关联的功能/项目/epic ID
doc_kind: note       # plan|discussion|research|bug-report|note
created: YYYY-MM-DD
---
```

然后写一个简单的脚本（或让 AI 写），统计你项目中 frontmatter 的覆盖率。

**提示词**：

> 帮我写一个脚本，遍历 docs/ 下所有 .md 文件，检查每个文件是否有 YAML frontmatter（以 `---` 开头和结束）。
> 输出：
> - 总文件数
> - 有 frontmatter 的文件数
> - 覆盖率百分比
> - 缺失 frontmatter 的文件列表

### 验收标准

- [ ] 至少 5 个文件补上了 frontmatter
- [ ] 有一个可运行的覆盖率统计脚本
- [ ] 覆盖率从审计前到审计后有明确提升

## 核心练习 3：用 7-slot 模板记录一个教训

### 背景

"注意不要"不是防护，"be careful"不是机制。7-slot 模板强制你把教训变成可执行的知识。

### 任务

回忆一个你在项目中踩过的坑（如果实在想不到，就用"AI 生成的代码有 bug 但测试全过"这个万能坑），用 7-slot 模板记录：

```markdown
### LL-001: <标题>

- 坑：<一句话：踩了什么坑>
- 根因：<为什么会踩>
- 触发条件：<什么条件下会复发>
- 修复：<当时怎么修>
- 防护：<可执行机制——具体的规则/测试/脚本>
- 来源锚点：<commit hash / 文件路径 / 聊天记录截图>
- 原理（可选）：<第一性原理>
```

### 验收标准

- [ ] 7 个字段（原理可选）都填写了
- [ ] "防护"字段是可执行的（不是"注意"/"小心"/"记得"）
- [ ] "来源锚点"可追溯（能找到原始证据）
- [ ] 给一个不了解背景的人看，他能理解这个坑并知道如何避免

## 进阶挑战（选做）

### 挑战 1：设计你的三层记忆架构

参考我们的热/温/冷三层，为你自己的项目设计一套记忆架构：

1. **热层**：什么信息需要随时可见？（类比我们的 BACKLOG.md）
2. **温层**：什么信息需要"一个入口就找到"？（类比 Feature 聚合文件）
3. **冷层**：什么信息需要保留但不必日常浏览？（类比归档）

考虑：
- 你的项目用什么工具管理任务？（GitHub Issues / Linear / Notion / ...）
- "热层"是不是已经有了？（比如看板/sprint board）
- "温层"缺不缺？（能不能一个入口看到某个 epic 的全貌？）
- "冷层"有没有归档机制？（完成的任务是消失了还是留着碍事？）

### 挑战 2：把一个经验编码成 Skill（或 Custom Instruction）

选一个你反复做的事情（比如"写 commit message"、"做 code review"、"处理 CI 失败"），把它从"脑子里的经验"变成"可执行的 Skill"：

1. **写 description**：Use when / Not for / Output 三件套
2. **写 workflow**：分步骤，每步有明确的输入/输出
3. **写 guard**：什么情况下这个 Skill 应该拒绝执行（防止误触发）

不一定要用 Cat Café 的 Skill 格式——Claude Code 的 Custom Slash Command、Cursor 的 .cursorrules、甚至一个 Makefile target 都可以。关键是把隐性经验变成显性的、可执行的、可传承的知识。

**判断标准**：如果你把这个 Skill 给一个新加入项目的人（或新起的 AI session），他/它能不能不问你就做对这件事？如果能——恭喜，你完成了从"经验"到"知识"的转化。

## 自检

完成以上练习后，问自己：

> "三个月后如果有人问我'这个项目 X 功能什么情况'，我能在 30 秒内回答吗？"

如果答案是"能"，你的知识管理体系就够用了。
