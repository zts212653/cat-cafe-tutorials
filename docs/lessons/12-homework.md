# 第十二课作业：设计你的去中心化协作层

## 核心练习

### 练习 1：中心编排的单点偏见实验

**目标**：亲手体验中心节点偏见如何传播到全系统。

1. 选两个不同的 LLM（比如 Claude + GPT，或任意两个你能调用的模型）
2. 准备一个有争议的技术选型问题，比如：
   > "做一个 multi-agent 的消息路由，应该用 Redis Pub/Sub 还是用 HTTP 轮询？"
3. **中心模式**：让模型 A 做 "boss"，先让 A 给出结论，再把 A 的结论发给模型 B，问 B 是否同意
4. **对等模式**：让 A 和 B 各自独立回答同一个问题（不给对方的上下文），然后你来对比两份答案
5. 填写对比表：

| 维度 | 中心模式（B 看过 A 的结论） | 对等模式（AB 独立回答） |
|------|--------------------------|----------------------|
| B 的结论和 A 相同吗？ | | |
| B 提出了 A 没有的论点吗？ | | |
| 你觉得哪种方式信息量更大？ | | |

**检查点**：在中心模式下，B 是否表现出了明显的锚定效应——倾向于在 A 的框架上微调，而不是独立推导？

### 练习 2：最小跨模型 Review

**目标**：验证不同模型的盲点差异是否真能找到更多问题。

1. 用模型 A 写一段代码（100-200 行），比如：
   - 一个简单的 webhook 处理器
   - 一个消息队列的消费者
   - 一个用户认证中间件
2. 让**同一个模型 A** review 这段代码，记录发现的问题数量和类型
3. 让**另一个模型 B** review 同一段代码，记录发现的问题数量和类型
4. 对比：

| 维度 | 模型 A 自审 | 模型 B 交叉审 |
|------|-----------|-------------|
| 发现问题总数 | | |
| 逻辑错误 | | |
| 安全问题 | | |
| 设计缺陷 | | |
| 只有一方发现的问题 | | |

**检查点**：模型 B 是否找到了至少一个模型 A 自审遗漏的问题？如果是，这就是跨家族 review 的价值。

### 练习 3：设计你的两层架构

**目标**：为你自己的场景设计"对等判断层 + 结构化执行层"。

1. 描述你的场景（比如：一个翻译团队、一个代码仓库维护团队、一个内容审核系统）
2. 画出你的对等判断层：
   - 有几个 agent？各自什么角色？
   - 谁可以质疑谁？（全对等？还是有限制？）
   - 分歧怎么收敛？（投票？证据比拼？升级到人类？）
3. 画出你的结构化执行层：
   - 用什么做任务队列？
   - 每个 agent 的执行环境怎么隔离？
   - 有哪些护栏（硬约束）？
4. 问自己：如果去掉其中一个 agent，系统会崩溃还是降级？

**检查点**：你的设计是否做到了"去掉一只猫，降级但不崩"？如果去掉任何一个 agent 系统就完全停摆，说明你的设计还有单点依赖。

## 进阶挑战

### 挑战 1：模拟三阶段冲突解决

实现一个最小的冲突解决流程：

```python
# 伪代码框架
class PeerAgent:
    def __init__(self, name: str, model: str): ...
    def independent_judge(self, question: str) -> dict:
        """独立判断，返回 {stance, reasoning, evidence}"""
        ...

class ConflictResolver:
    def __init__(self, agents: list[PeerAgent]): ...

    def phase1_independent(self, question: str) -> list[dict]:
        """第一阶段：所有 agent 独立判断"""
        ...

    def phase2_cross_calibrate(self, judgments: list[dict]) -> list[dict]:
        """第二阶段：每个 agent 看到其他人的判断后，更新自己的立场"""
        ...

    def phase3_converge(self, updated: list[dict]) -> dict:
        """第三阶段：检查是否收敛。未收敛则升级到人类"""
        ...
```

要求：
- `phase1` 中每个 agent 不能看到其他 agent 的输出
- `phase2` 中每个 agent 必须明确回应其他人的论点（不能说"都行"）
- `phase3` 中如果仍有分歧，返回 `{converged: false, escalate_to: "human"}`

### 挑战 2：写五条你的系统的 Review 铁律

参考我们的 LL-003 教训（"review 变成礼貌性同意"），为你的多 agent 系统写五条 Review 规则：

1. 每条规则必须包含：**规则本身** + **为什么需要这条规则**（即背后的风险）
2. 至少有一条关于"禁止说什么"（比如我们禁止说"修不修都行"）
3. 至少有一条关于"必须做什么"（比如我们要求跨家族 review）
