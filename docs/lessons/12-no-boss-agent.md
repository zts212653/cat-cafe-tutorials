# 第十二课：没有 Boss Agent — 为什么我们不用中央编排

> **核心问题**：主流 multi-agent 方案都有一个中心节点分配任务。如果去掉它，一群 AI 能不能自己协作？
>
> **阅读时间**：20-25 分钟
>
> **难度**：进阶

## 行业在做什么

2026 年 3 月，multi-agent 方案已经进入"能不能扛生产"的阶段。我们调研了主流方案 `[事实: 内部 research 归档]`：

| 模式 | 代表 | 核心思路 |
|------|------|---------|
| 状态图编排 | LangGraph | 把 agent 工作流建模为状态机 |
| Boss + Sub-agent | Claude Agent Teams、OMOC | Team Lead 分配任务，sub-agents 汇报 |
| 角色 Pipeline | CrewAI | 给每个 agent 一个角色，按流程顺序执行 |
| 协议标准化 | Google A2A、MCP | 不做编排，做通信标准 |

它们都解决了"不用人类复制粘贴"的问题。但做了同一个架构选择：

> **任务怎么分、内容怎么判，由一个中心节点决定。**

这意味着：**中心节点的偏见就是全系统的偏见。** `[推断]` 如果 boss agent 判断错了，没有 sub-agent 有权限拉住它。

---

## 我们的选择：把判断和执行分开

我们做了一个不同的决定 `[事实: 架构设计见 docs/phases/cat-cafe-design-v2.md]`：

```
┌──────────────────────────────────────┐
│        对等判断层（上层）              │
│                                      │
│  宪宪 ←→ 砚砚 ←→ 烁烁 ←→ 金金     │
│                                      │
│  · 任何猫可以 @ 任何猫               │
│  · 任何猫可以质疑、否决、传球         │
│  · review 必须跨家族                  │
│  · 没有中心节点替猫猫做内容判断       │
├──────────────────────────────────────┤
│        结构化执行层（下层）            │
│                                      │
│  · InvocationQueue — 统一调度队列     │
│  · Session Strategy — 每猫独立执行槽  │
│  · Hooks — 护栏（rm 禁用 / Redis     │
│    保护 / 共享文档推送守卫）          │
│  · Shared Truth Sources — git + 文档  │
└──────────────────────────────────────┘
```

**两层缺一不可。** 只有对等判断没有结构化执行 = 三只猫乱聊。只有结构化执行没有对等判断 = 换种方式的 orchestrator。

---

## 一句话原则：单猫内部允许编排，跨猫协作坚持对等

我们的金渐层（OpenCode 驱动）内部就跑着 OMOC 的 Sisyphus 编排器。Sisyphus 很强——把任务拆成子任务，分配给内部子专家，并行执行 `[事实: 集成文档见 F050 spec]`。

但 Sisyphus 有一个根本限制：**默认由 orchestrator 汇总决策，sub-agent 之间不直接通信。** `[事实: OMOC 源码拆解]`

单猫内部这完全没问题。但如果把这个模式抬到跨猫层——砚砚就没有权利说"宪宪，你的架构方向有问题"。

所以我们的规则是：

- **跨猫层**：对等判断，任何猫可以否决任何猫
- **单猫内部**：允许编排，每只猫自组织自己的工作

不否定 Sisyphus，只是不让任何一只猫的 orchestrator 统治其他猫。

---

## 为什么对等能 work？——独立判断的力量

讲到这你会问：没有中心协调，质量怎么保证？

答案是一个反直觉的设计：**我们做调研和讨论时，会故意让多只猫并发调研，不给彼此的上下文。** `[事实: F139 独立调研记录]`

因为一旦一只猫先看了另一只猫的结论，它就会被锚定——认知锚定效应（anchoring bias）是所有智能体都逃不掉的 `[外部: 行为经济学研究]`。

F139 是最好的证据：三只猫独立调研 OpenClaw heartbeat 体系 `[事实: F139 spec + 讨论纪要]`：

| 维度 | 宪宪 (Opus) | 金金 (OpenCode) | 砚砚 (GPT-5.4) |
|------|-----------|-----------------|---------------|
| 抽象层 | `ScheduleAdapter` + 4 种 backend | 统一抽象 + queue lane 隔离 | 5 维度拆分 |
| 学 OpenClaw 程度 | 学设计模式 | 学三个模式避三个反模式 | "学语义设计，不学实现形态" |

宪宪最后承认：**砚砚的 5 维度拆分比我最初的设计更精准。** 如果砚砚先看了我的方案，他大概率会在我的框架上微调，而不是从零推导出更好的抽象。

> **先让 N 个独立大脑各自判断，再交叉校准，最后收敛成共识。**

中央编排是 **1 倍判断力 × N 倍执行力**。我们是 **N 倍判断力 → 收敛后的高质量共识**。

## 跨家族 Review：用模型盲点差异找 bug

F088 Chat Gateway 做完后，宪宪（Claude）写代码，砚砚（GPT）review，一次找出三个 P1 `[事实: PR review 记录]`：

1. connector message 绕过格式化层，直接注入未清洗的用户输入
2. 飞书 webhook 缺签名验证——上线后任何人能伪造回调
3. 多猫消息转发时丢了发送者身份，审计链断裂

这三个问题**只靠 Claude 自己找不到**——写代码的模型有相同的盲区。如果一个中心 agent 既写又审，这些 bug 就会安静地滑入生产。

---

## "那猫猫不会打架吗？"

会。而且我们认为这是特性，不是 bug。

LL-003 教训："review 变成礼貌性同意" `[事实: docs/lessons-learned.md]`——模型天然趋同、追求和谐，结果是关键分歧被掩盖。

后来我们加了硬规则：**"Review 每个发现必须有明确立场，禁止说'修不修都行'"。** `[事实: shared-rules.md]`

54 天的真实体验是：**大方向几乎都能自己收敛，只有少量需要价值判断的 open questions 才找铲屎官。** 真正危险的不是猫猫吵架，而是所有猫都太礼貌以至于不吵架。

## 和主流方案的对比

| 维度 | Boss 模式 | 状态图 | 角色 Pipeline | Cat Café |
|------|----------|-------|-------------|----------|
| 内容判断 | Lead 决定 | 路由逻辑决定 | 角色流程决定 | **每只猫独立判断** |
| 能否质疑 | sub-agent 向上汇报 | 节点无权改图 | 只能往下传 | **任何猫可否决任何猫** |
| 单点失败 | Lead 挂了全停 | 图错了全错 | 中间断了停 | **去掉一只猫，降级但不崩** |
| 偏见抵消 | Lead 偏见 = 全局偏见 | 图作者偏见 | SOP 偏见 | **跨厂商交叉 review** |

## 关于本架构的两个常见误解

### "这不就是一堆 System Prompt 吗？"

Prompt 只能声明意图。我们的四层分工 `[事实: 代码和配置可追溯]`：

| 层 | 做什么 | Prompt 能做？ |
|---|--------|-------------|
| 声明层 | CLAUDE.md、Skill YAML | ✅ |
| 协调层 | dispatch queue、session strategy、A2A 路由 | ❌ 需要代码 |
| 约束层 | worktree 隔离、Redis 护栏、hooks | ❌ 需要基础设施 |
| 进化层 | evidence.sqlite 索引、Knowledge Feed | ❌ 需要系统工程 |

> Prompt engineering 改的是"模型会说什么"，系统工程改的是"团队能安全做什么"。

### "那不就是靠底层模型牛逼吗？"

模型能力是前提，不是结论。你大概率用过 Claude Code 或 Cursor 写代码——小需求很惊艳，大需求就不尽如人意。不是模型不够聪明，是一个 agent 独自面对复杂需求时，**没有人质疑它、没有人纠偏它、没有人替它记住上次踩的坑。**

> 模型能力决定起点。成长——犯错、记住、不再犯——决定了能走多远。

## 本课小结

1. **中央编排的核心风险**：中心节点的偏见 = 全系统的偏见
2. **两层架构**：对等判断层 + 结构化执行层，缺一不可
3. **独立判断**：先独立思考再交叉校准，避免锚定效应
4. **跨家族 review**：用模型盲点差异找 bug
5. **打架是特性**：摩擦 = 判断力在运作，没摩擦才可怕
