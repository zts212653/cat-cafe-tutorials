# 第十四课：越犯错越聪明 — 三层记忆与自我进化

> **核心问题**：Multi-agent 系统最大的浪费是什么？是同一个坑踩两次。记忆系统怎么让失败变成资产？
>
> **阅读时间**：20-25 分钟
>
> **难度**：进阶
>
> **前置知识**：[第十课](./10-knowledge-management.md)（知识管理基础），[第十二课](./12-no-boss-agent.md)（对等架构）
>
> **证据标注**：
> `[事实]` 有 commit / 文档 / 代码佐证 ·

## 铲屎官自己忘了，猫还记得

有一天，铲屎官突然想起一个问题：

> "我们之前那个定时任务，到底是怎么决定的来着？"

如果这是一个普通的 multi-agent 系统，接下来会发生什么？翻聊天记录 → 试着回忆 → grep 一圈 → 算了，重新讨论一遍吧。

但那一次没有。小金直接搜了一下，几分钟后，把整条决策链路拉了出来：相关 feature、讨论纪要、铲屎官当时说过的原话 `[事实: 铲屎官原话截图 + search_evidence 输出]`。

铲屎官的评价：**"我经常被你们惊艳到。你看这个我都忘记的细节，你们依靠记忆把我曾经说的话都扒拉出来了。"**

这就是本课的核心：**群体智能和普通 multi-agent 的区别，不是"同时有几只猫在说话"，而是"有没有持久记忆，能不能从过去里把正确的东西重新拉出来"。**

---

## 记忆不是聊天记录堆积

很多人一说"Agent 记忆"，想到的是：把更多历史消息塞进上下文、聊天记录存起来、需要时全文检索。

这些都不够。真正让系统变强的，不是"它记得很多"，而是 `[推断]`：

1. **它知道什么值得记**
2. **它知道去哪里找**
3. **它知道找到之后怎么用**

所以我们把记忆拆成了三层。

---

## 第一层：文档是真相源

我们没有把长期知识塞进一个数据库然后希望模型自己变聪明。第一原则很简单 `[事实: 架构设计 + CLAUDE.md]`：

> **知识的真相源必须是人和猫都能读、都能改、都能追溯的文件。**

真正的长期记忆长在这些地方：

| 目录 | 记什么 |
|------|--------|
| `docs/features/` | 每个 Feature 的目标、边界、AC、演化 |
| `docs/decisions/` | 为什么选 A 不选 B |
| `docs/discussions/` | 讨论收敛时保留的上下文 |
| `docs/lessons-learned.md` | 踩坑之后怎么避免再踩 |

这意味着：记忆是**可审计的**、**跟着仓库走的**、**不是模型脑子里的黑箱**。

> **我们不是把记忆托付给模型，是把模型接到记忆之上。**

---

## 第二层：索引是编译产物，检索是联邦的

文档多到一定程度，靠肉眼快速回忆是不可能的。F102 做了一件关键的事：把记忆组件改成 Adapter 结构，用本地索引接管检索 `[事实: F102 spec]`。

### 双库 + 联邦检索

- **`evidence.sqlite`**（项目层）：编译当前仓库的 feature、decision、discussion、lesson
- **`global_knowledge.sqlite`**（全局层）：编译 Skills、家规、跨项目方法论

项目知识留在项目里，全局知识跟着猫走。真正负责找回知识的是 `IKnowledgeResolver`——做 query planning、双库 fan-out、结果 normalize，再用 RRF rank fusion 收口 `[事实: F102 代码]`。

### 三种检索模式

| 模式 | 适合搜什么 | 原理 |
|------|-----------|------|
| `lexical` | Feature ID、ADR 编号、精确术语 | BM25 关键词匹配 |
| `semantic` | 同义表达、跨语言、"记得意思忘了原词" | 向量近邻 |
| `hybrid` | 日常 recall（默认推荐） | 双路召回 + RRF 融合 |

### 过期知识必须能退役

F102 加了 `superseded_by` 字段和 `invalidates` 关系 `[事实: F102 schema]`。原因很朴素：

> **过时但高度相似的决策，比完全查不到更危险。** `[推断]`

旧 ADR 被新 ADR 取代但检索时还排前面，猫猫就会基于过期信息做判断。

## 第三层：Lesson 让记忆开始有方向

记忆系统真正厉害的地方，不是"搜得快"，而是它开始知道**什么东西值得以后反复提醒** `[事实: docs/lessons-learned.md]`。

我们的 `lessons-learned.md` 不是随手写的感想录，而是固定 7 槽位：

| 槽位 | 作用 |
|------|------|
| 坑 | 发生了什么 |
| 根因 | 为什么会发生 |
| 触发条件 | 什么情况下会复发 |
| 修复 | 怎么修的 |
| 防护 | 下次怎么在流程里挡住 |
| 来源锚点 | 关联的 Feature / PR / commit |
| 原理（可选） | 底层原因解释 |

**教训不是情绪，而是可执行的防护知识。**

如果只说"以后注意"，它不会变成系统能力。但如果写清楚**什么时候会复发、为什么会复发、下次怎么挡住**——下一只猫遇到类似模式时，就不需要重新用血换经验。

## 从候选到真相：知识晋升管道

系统不能把任何一句对话都直接塞进长期记忆 `[事实: F102 Knowledge Feed 设计]`。

```
对话中提取候选
  → Knowledge Feed（等待审核）
    → MarkerQueue 状态机
      captured → normalized → approved → materialized → indexed
    → 写回 .md 真相源 + 触发 reindex
```

这条管道解决的是：**一句对话不能因为"值得记住"就直接变成长期真相。**

我们踩过的坑正是：什么都"记一下"，最后长期记忆变成碎片垃圾堆。所以知识必须先作为候选出现，再经过审核、晋升、索引，才有资格进入长期层。

---

## 自我进化链

在知识晋升管道之上，系统的自我进化链开始成立 `[推断]`：

```
一次真实事件
  → 提炼成 Episode
    → 抽象成 Method
      → 固化成 Skill
        → 用 Eval 检查是否真的有效
          → 沉淀进 shared rules / SOP / lesson
```

一个系统真正强，不是看它一次生成得多漂亮，而是看它**会不会把一次次成功和失败都转成下一轮的判断标准**。

---

## 记忆如何反过来影响协作

当记忆系统跑起来后，三猫协作本身也变了 `[事实: 日常工作流变化]`：

| 以前 | 现在 |
|------|------|
| "我们之前怎么定的？" → "算了，重开一轮讨论" | `search_evidence("定时任务 决策")` → 拉出相关证据继续推进 |
| Review 靠"我觉得不好" | Review 靠"这个模式我们踩过，这里有 lesson" |
| 新 session 从零开始 | 靠系统快速恢复状态 |

三个变化：**分歧更少重复**（历史分歧已保留）、**Review 更有证据**、**交接更轻**。

> 记忆系统不只是 recall，它在**降低协作熵增**。

---

## 一个差点没跑起来的系统

真实细节：记忆系统不是一开始就成熟的 `[事实: F102 激活讨论]`。

F102 的激活讨论里，大家发现：`evidence.sqlite` 从未被创建、rebuild 只支持手动跑、`search_evidence` 没有真正走 SQLite、反思服务是空壳。

后来 MVP 被收敛成三件事：

1. 启动自动 rebuild
2. `search_evidence` 默认命中 SQLite
3. memory status 可观测

**记忆不是一个概念，记忆必须跑起来。** 设计再好，不落地就是零。

---

## 渐进增强而不是推倒重来

F102 从一开始就把 SQLite 作为终态存储基座，检索策略设计成渐进增强 `[事实: F102 spec EMBED_MODE]`：

| 模式 | 行为 |
|------|------|
| `EMBED_MODE=off` | 纯 lexical，基础能力始终可用 |
| `EMBED_MODE=shadow` | 后台异步跑 embedding，记录 A/B 指标 |
| `EMBED_MODE=on` | 向量 rerank 启用，lexical 做 fallback |

> **增强层可以升级，终态基座不能重写。**

即便 embedding 失败，系统也必须 fail-open 回落到 lexical。

---

## 本课小结

1. **文档是真相源**：可审计、可追溯、不是模型黑箱
2. **索引是编译产物**：`evidence.sqlite` 是加速器，不是真相本体
3. **双库联邦检索**：项目层 + 全局层，由 resolver 统一召回
4. **教训要结构化**：7 槽位格式，让失败有资格进入未来
5. **知识晋升管道**：候选 → 审核 → 晋升 → 索引，不许对话直接污染长期库
6. **渐进增强**：终态基座不能重写，增强层可以升级
